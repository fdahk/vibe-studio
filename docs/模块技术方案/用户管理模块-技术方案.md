# 用户管理模块 · 技术方案

> 身份与账户 · 多方式登录注册 · 鉴权传输与会话 · 用户资料

- **模块**：用户管理（user）
- **状态**：设计评审中（部分已落地，鉴权传输/会话为本轮新增）
- **日期**：2026-06-20

---

## 1. 模块概述与职责

用户管理模块是整个平台的**鉴权地基**——所有其它模块的请求都经它鉴权。它负责：

1. **身份与账户**：用户主体 + 多种登录凭证。
2. **登录 / 注册**：用户名密码、手机号验证码、GitHub OAuth（预留 Google/Apple）。
3. **鉴权传输与会话**：access/refresh 双 token、会话轮换/吊销、静默续期。
4. **用户资料读取**：当前登录用户信息。

**边界（不在本模块）**：权限/RBAC、组织/团队、用户资料编辑（改昵称/头像）——各自独立演进。

## 2. 总体架构（DDD 分层）

```
api  (HTTP handler + Auth 中间件 + openapi 契约)
  ↓
application  (Service：注册/登录/发码/手机登录/OAuth/续期/登出/查用户)
  ↓                       ↑ 实现接口
domain  (User/Identity/Session 实体 + Repository 接口；纯领域、无框架)
  ↑ 实现接口
infra  (persistence GORM · sms 可插拔发送器+Redis验证码 · oauth GitHub客户端+state)
```

依赖方向 `api → application → domain ← infra`；`domain` 禁止 import 任何框架/DB。

## 3. 数据模型（三张表）

关系：一个 `user` 拥有多个 `identity`（登录方式）和多个 `session`（登录设备）。

```
user 1 ── N identity      (provider, provider_uid) 唯一
user 1 ── N session       (一台设备一行)
```

### 3.1 `users`（身份主体）
`id, username(uniq), email, phone(idx), nickname, avatar, status, created_at, updated_at`
**不含任何登录凭证**——凭证在 identities。

### 3.2 `identities`（登录凭证，多 provider）
`id, user_id(idx), provider, provider_uid, secret, created_at, updated_at`
`(provider, provider_uid)` 唯一。
- `local`：provider_uid=用户名，secret=bcrypt 密码哈希
- `phone`：provider_uid=手机号，secret 空
- `github`/`google`/`apple`：provider_uid=第三方 id，secret 预留

### 3.3 `sessions`（登录会话 / refresh）— 本轮新增
`id, user_id(idx), token_hash(uniq), prev_token_hash, user_agent, ip, expires_at, revoked_at, last_used_at, created_at, updated_at`
一行 = 一台设备的登录；`token_hash` = 当前有效 refresh 的 SHA-256。

## 4. 多 provider 鉴权架构

**核心**：身份主体（user）与登录方式（identity）**分离**——一个 user 可绑多个 identity，任一 provider 登录都解析到同一 user。

**决策：users + identities 分表 vs 单表加 provider 列**
- 单表加 `provider`/`provider_uid` 列：一个用户多种登录方式时要么多行重复用户资料、要么塞不下。
- **分表（选）**：联合身份业界标准；加 Google/Apple 只多一条 identity、user 主体不动。✓

## 5. 登录 / 注册方式详解

### 5.1 用户名密码（local）— 已落地
- **注册**：查重（GetIdentity local/username）→ bcrypt 哈希 → 事务建 user+local identity → 签发。
- **登录**：查 identity → bcrypt 校验 → 签发。
- **安全**：bcrypt（自带 salt）；登录失败统一报"用户名或密码错误"，不泄露账号是否存在。

### 5.2 手机号验证码（phone）— 已落地
- **发码**：校验手机号格式 → Redis 冷却闸（`SET NX` 60s 防刷）→ `crypto/rand` 6 位码 → 存 Redis（短 TTL 5min）→ `Sender` 下发。
- **登录**：校验码（错误计数超 5 次作废防爆破；常量时间比较防时序侧信道）→ 命中 phone identity 则登录，否则**自动注册** → 签发。
- **`Sender` 可插拔**：dev = `ConsoleSender`（验证码打日志）；生产 = 火山引擎/阿里云适配器（待云账号 + 签名/模板审核，只换实现、业务不动）。
- **决策**：① 短信网关可插拔——签名审核有门槛、不该阻塞开发；② 验证码用 Redis——快、TTL 原生、不压 DB；③ `crypto/rand` + 常量时间比较——防预测与时序攻击。

### 5.3 GitHub OAuth（github）— 已落地
- **流程**：`GET /auth/oauth/github`（生成 state 存 Redis、跳 GitHub）→ `callback`（校验 state → 后端用 client_secret 换 token → 拉用户 → find-or-create → 签发）。
- **安全**：client_secret **仅在后端**；state 单次使用防 CSRF；回跳改为只种 cookie、URL 不带 token。
- **通用 `LoginByOAuth`**：按 `(provider, provider_uid)` find-or-create，Google/Apple 直接复用。
- **决策：首个第三方选 GitHub 而非微信**——微信网站/App 登录要企业营业执照 + ¥300 认证（个人办不了）；GitHub 个人免费零门槛，且本平台用户就是开发者、天然有 GitHub。

### 5.4 预留 Google / Apple
provider 常量与 `LoginByOAuth` 已就位，加各自 OAuth 客户端即可。

## 6. 鉴权传输与会话（本轮新增 / Spec 1）

### 6.1 token 模型

| 凭证 | 形态 | 时长 | 存储 | 用途 |
|---|---|---|---|---|
| access | JWT（无状态） | 15min | 前端**内存**（Zustand，不持久化）；响应体返回 | API `Bearer` |
| refresh | **不透明随机串** | 30 天 | **HttpOnly+Secure+SameSite cookie**；服务端存 SHA-256 | 续期/登出 |

### 6.2 轮换 + 复用检测
- 每次 `/refresh` 换发新 refresh：`token_hash` 换新，旧值进 `prev_token_hash`（容一次轮换宽限，兼顾客户端并发重试）。
- 来的 refresh 命中 `prev_token_hash`（已轮换又被重放）→ 判泄露 → 吊销整条会话 → 强制重登。

### 6.3 端点

| 方法 路径 | 行为 |
|---|---|
| `register` / `login` / `login/phone` | 成功：建 session + 种 refresh cookie + 响应体返回 `access_token`(+user) |
| `oauth/github/callback` | 建 session + 种 cookie + **跳回 `/oauth/callback`（URL 不带 token）** |
| `POST /auth/refresh` | 读 cookie→校验 session→轮换→重种 cookie→返回新 access |
| `POST /auth/logout` | 读 cookie→吊销 session→清 cookie |
| `GET /users/me` | 不变（Bearer access） |

### 6.4 Cookie / CSRF / CORS
- Cookie：`HttpOnly`（JS 不可读防 XSS）、`SameSite=Lax`（防 CSRF，且允许 OAuth 顶级回跳携带）、`Secure`（生产强制、dev 配置关）、`Path=/api/v1/auth`（只在 refresh/logout 携带）、`Max-Age=30d`。
- CSRF：SameSite=Lax + cookie Path 限定 + access 走 Bearer 自定义头；`/refresh`/`/logout` 仅 POST。
- CORS：带 credentials 后禁止 `*`，改回显白名单具体源 + `Allow-Credentials: true`（新增 `ALLOWED_ORIGINS`）。

### 6.5 前端方案
- access 只存内存（删 localStorage 持久化）。
- 启动静默续期：加载时 access 空 → 调 `/refresh` 恢复登录态。
- openapi-fetch 中间件：附 Bearer；401 自动 `/refresh` 重试，再失败清态跳登录；`credentials:'include'`。
- 登出调 `/logout`；OAuth 回调页改为落地后调 `/refresh`。

## 7. 关键决策汇总（含否决备选）

| 决策项 | 选择 | 否决的备选 | 理由 |
|---|---|---|---|
| 身份建模 | users + identities 分表 | 单表加 provider 列 | 多 provider 正解、加第三方不动主体 |
| access 存储 | 前端内存 | localStorage / cookie | 防 XSS 持久窃取、暴露窗口小 |
| refresh 存储 | DB `sessions` 表 | Redis / 无状态 JWT | durable + 可即时吊销 + 设备管理 |
| refresh 格式 | 不透明随机串(存哈希) | 自包含 JWT | 可即时吊销/轮换/复用检测 |
| 验证码存储 | Redis(短 TTL) | DB | 快、TTL 原生、不压 DB |
| 短信网关 | 可插拔 + dev console | 直接接某网关 | 签名审核不阻塞开发 |
| 首个第三方 | GitHub | 微信 | 个人零门槛 + 用户就是开发者 |

## 8. 安全性分析（全模块）

| 威胁 | 缓解 |
|---|---|
| 密码泄露 | bcrypt 哈希、仅存哈希 |
| 验证码爆破/预测 | crypto-rand + 5 次作废 + 常量时间比较 + 冷却频控 |
| XSS 持久窃取 token | access 仅内存且短命、refresh HttpOnly |
| CSRF | SameSite=Lax + Path 限定 + Bearer 自定义头 |
| refresh 盗用/重放 | 轮换 + 复用检测吊销整条会话 |
| 账号枚举 | 登录失败统一错误 |
| 库泄露 | 密码/验证码/refresh 均存哈希或短时；JWT secret 仅服务端 |
