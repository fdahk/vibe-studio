# Hertz（CloudWeGo）—— Go HTTP 框架选型

> 本文是一篇"技术选型理由"教学文档。面向想深入理解 Go HTTP 框架选型本质的读者。
> 全文区分三类动机：**纯技术动机**（性能/功能）、**生态/dogfooding 动机**（字节强烈使用自家 CloudWeGo 生态）、**平台/私有化动机**（coze 是部署给上万企业的私有化平台）。
> 凡是"确定事实"都给出 coze-studio 源码/版本佐证；凡是"合理推断的动机"会明确标注；不确定的数字标注"待核实"。

---

## 1. Hertz 是什么（零基础扫盲）

**一句话**：Hertz 是字节跳动开源的、面向高并发场景的 Go 语言 **HTTP 框架**，属于字节 **CloudWeGo** 开源微服务全家桶的一员。它的定位等价于 Go 生态里的 Gin/Echo，但底层网络 IO 不走 Go 标准库 `net/http`，而是走字节自研的网络库 **netpoll**。

把概念拆开讲：

- **什么是"HTTP 框架"**：你写一个 Web 后端，最基础可以只用标准库 `net/http`——它给你一个 `http.Server`、一个 `http.Handler` 接口，剩下路由匹配、参数绑定、中间件、JSON 序列化、错误处理全靠你自己拼。"框架"就是把这些重复劳动封装好：给你**路由树**（`GET("/api/x", handler)`）、**中间件链**（`Use(...)`）、**请求上下文对象**（一个 `Context` 把请求/响应/参数/状态都装一起）、**参数绑定与校验**（自动把 query/body 映射到 struct）。Gin、Echo、Hertz 干的都是这件事。

- **什么是 netpoll**：Go 标准库的网络模型是"**一个连接一个 goroutine**" + 阻塞式 `Read/Write`，由 Go runtime 的 netpoller 在底层用 epoll/kqueue 做就绪通知。这个模型**极其好写**（同步代码读起来像阻塞 IO），但在**海量长连接 / 超高 QPS** 下有两个代价：① 每连接一个 goroutine，连接数上去后调度和栈内存有压力；② 标准库的 buffer 设计会带来较多内存拷贝。netpoll 是字节自研的 **基于 epoll 的多路复用网络库**，核心卖点是 **主动式 epoll 事件循环**（不是一连接一 goroutine 死等）、**零拷贝 buffer**（`nocopy` 的 LinkBuffer，读写尽量不复制内存）、**连接级别的内存池复用**。Hertz 默认就跑在 netpoll 上。

- **CloudWeGo 是什么**：字节开源的一套微服务中间件全家桶，主要成员：
  - **Hertz** —— HTTP 框架（本文主角）
  - **Kitex** —— RPC 框架（Thrift/gRPC/protobuf，服务间内部调用）
  - **netpoll** —— 上面两者共享的底层网络库
  - **Eino** —— LLM 应用编排框架（构建 Agent / RAG / workflow 的 Go 版"LangChain"）
  - 还有 hertz-contrib / kitex 周边扩展生态

  对 coze-studio 这种 "Go 写的 AI Agent 平台" 来说，CloudWeGo 几乎是为它量身定做的拼图：HTTP 入口用 Hertz，LLM 编排用 Eino，底层网络用 netpoll，全是同一个团队、同一套设计哲学。

- **代码生成工具 hz**：Hertz 配套一个命令行工具 `hz`，能从 **IDL**（Thrift/Protobuf 接口定义文件）一键生成路由注册、handler 骨架、请求/响应 model。生成的文件头部会带 `// Code generated by hertz generator`。这是后面"在 coze 里干什么"一节的核心证据。

> 心智模型：**Hertz ≈ "Gin 的 API 手感" + "字节自研 netpoll 的性能底座" + "和 Kitex/Eino 同源的生态归属"**。它没有重新发明 Web 框架的概念，发明的是底下那层网络引擎和上面那套代码生成 + 生态协同。

## 2. 在 coze-studio 里干什么（读源码举证）

以下均以本地源码 `/tmp/coze-studio` 为准，版本以 `backend/go.mod` 为准。

### 2.1 版本与依赖（确定事实）

`backend/go.mod`（module `github.com/coze-dev/coze-studio/backend`，`go 1.24.0`）里和 Hertz/CloudWeGo 相关的依赖：

```
github.com/cloudwego/hertz v0.10.2
github.com/hertz-contrib/cors v0.1.0
github.com/hertz-contrib/sse  v0.1.0
github.com/cloudwego/eino    v0.4.8
github.com/cloudwego/netpoll v0.7.0   // indirect
github.com/cloudwego/gopkg   v0.1.4   // indirect
github.com/cloudwego/base64x v0.1.6   // indirect
```

几个**值得注意的事实**：

1. **netpoll 是 indirect 依赖**——它不是 coze 直接 import 的，而是被 Hertz 间接拉进来的。这印证了"选 Hertz 就等于选了 netpoll 底座"，开发者无需直接接触网络层。
2. **coze-studio 开源版里没有 Kitex**。`grep -ic kitex backend/go.mod` 结果为 0。这是一个**必须诚实指出的反直觉点**：很多人（包括本文写作前的预设）以为"既然字节用 CloudWeGo，coze 一定 Hertz+Kitex 一起上"。实际开源版后端是一个**单体 HTTP 服务**，服务内部用普通 Go 函数调用而非 RPC，所以只需要 Hertz，不需要 Kitex。（字节**内部**的 coze 是否拆成多个 Kitex 微服务——属于合理推断，**待核实**，不能写成事实。）
3. **Eino 在场（直接依赖）**，说明 coze 确实吃了 CloudWeGo 的另一块拼图来做 LLM 编排。Hertz 和 Eino 同属 CloudWeGo，这是"生态归属"动机的实锤之一。

### 2.2 服务启动：main.go（确定事实）

`backend/main.go` 头部第 17 行就是 `// Code generated by hertz generator.`——整个项目脚手架由 Hertz 工具链生成。核心启动逻辑（`startHttpServer`，约第 61–105 行）：

```go
opts := []config.Option{
    server.WithHostPorts(addr),                  // 默认 ":8888"
    server.WithMaxRequestBodySize(int(maxSize)), // 默认 200MB
}
// 可选 TLS：useSSL == "1" 时 server.WithTLS(cfg)
s := server.Default(opts...)

// CORS（hertz-contrib/cors）
config := cors.DefaultConfig()
config.AllowAllOrigins = true
corsHandler := cors.New(config)

// 中间件链（顺序敏感，注释明确写了 "Middleware order matters"）
s.Use(middleware.ContextCacheMW())     // must be first
s.Use(middleware.RequestInspectorMW()) // must be second
s.Use(middleware.SetHostMW())
s.Use(middleware.SetLogIDMW())
s.Use(corsHandler)
s.Use(middleware.AccessLogMW())
s.Use(middleware.OpenapiAuthMW())
s.Use(middleware.SessionAuthMW())
s.Use(middleware.I18nMW())             // must after SessionAuthMW

router.GeneratedRegister(s)
s.Spin()
```

读出来的信息：

- 用的是 `server.Default(...)`（自带 recover 等默认中间件）而非 `server.New(...)`，没有手动覆盖 transport——意味着**默认就跑在 netpoll 引擎上**，coze 没有切回标准库 transport。
- 配置项体现了**私有化平台的需求**：`MAX_REQUEST_BODY_SIZE` 默认 200MB（知识库上传大文件）、`UseSSL`/`SSLCertFile`/`SSLKeyFile`（企业内网自带证书私有化部署）、`LISTEN_ADDR` 可配。这些都是"部署给上万企业"的平台特征，不是 demo 项目会写的东西。
- 中间件链是一条**鉴权 + 可观测 + 国际化**的完整责任链：上下文缓存 → 请求探针 → Host → LogID（trace 关联）→ CORS → 访问日志 → OpenAPI 鉴权 → Session 鉴权 → i18n。注释强制了顺序约束（如 i18n 必须在 SessionAuth 之后，因为要拿到用户语言偏好）。

### 2.3 路由：IDL 生成的注册树（确定事实）

`backend/api/router/register.go`（头部 `// Code generated by hertz generator. DO NOT EDIT.`）的入口：

```go
func GeneratedRegister(r *server.Hertz) {
    //INSERT_POINT: DO NOT DELETE THIS LINE!
    coze.Register(r)        // 业务路由（IDL 生成）
    staticFileRegister(r)   // 前端静态资源 + SPA fallback
}
```

`backend/api/router/coze/api.go`（同样 `DO NOT EDIT`）是 `hz` 从 IDL 的 `api.${HTTP Method}` 注解生成的全量路由树，用 **分组 + 分组级中间件** 组织：

```go
root := r.Group("/", rootMw()...)
{
    _api := root.Group("/api", _apiMw()...)
    {
        _admin := _api.Group("/admin", _adminMw()...)
        {
            _config := _admin.Group("/config", _configMw()...)
            {
                _basic := _config.Group("/basic", _basicMw()...)
                _basic.GET("/get",   append(_getbasicconfigurationMw(), coze.GetBasicConfiguration)...)
                _basic.POST("/save", append(_savebasicconfigurationMw(), coze.SaveBasicConfiguration)...)
            }
            // ... knowledge / model / ...
        }
    }
}
```

- 路由规模：`api.go` 里 `GET/POST/PUT/DELETE` 注册点约 **246 处**（`grep -cE` 统计），覆盖 agent / knowledge / plugin / workflow / conversation / passport 等十多个 service handler 文件（`backend/api/handler/coze/*_service.go`）。这是一个**大型平台**的体量，不是玩具。
- 每个分组都有一个 `xxxMw()` 占位（`backend/api/router/coze/middleware.go`，头部 `// Code generated by hertz generator`，里面默认 `return nil`，留给开发者填）。这是 `hz` 生成 + 人工接业务的**典型协作模式**：生成的文件 `DO NOT EDIT`，扩展点（中间件、handler 业务体）由人填。
- handler 签名统一是 `func XxxApi(ctx context.Context, c *app.RequestContext)`（见 `conversation_service.go`），这是 Hertz 的标准 handler 形态——**注意它是双参数 `(context.Context, *app.RequestContext)`，而 Gin 是单参数 `(*gin.Context)`**，这点在第 4 节对比里很关键。

### 2.4 流式输出：SSE（确定事实，且是"为什么不能随便换框架"的关键）

coze 是 AI 平台，LLM 回答必须**流式逐字吐**（Server-Sent Events）。证据：`hertz-contrib/sse` 是直接依赖，`backend/infra/sse/impl/sse/sse.go` 把它包成内部接口：

```go
import "github.com/hertz-contrib/sse"

type SSenderImpl struct{ ss *sse.Stream }
func NewSSESender(ss *sse.Stream) *SSenderImpl { return &SSenderImpl{ss: ss} }
func (s *SSenderImpl) Send(ctx context.Context, event *sse.Event) error {
    return s.ss.Publish(event)
}
```

agent 对话、workflow 运行的流式 chunk（`backend/application/conversation/build_chunk_event.go`、`backend/api/handler/coze/agent_run_service.go`）都走这条 SSE 通路。**这意味着流式能力深度绑定了 hertz-contrib 生态**——换框架不是换个路由库那么简单，连流式输出层都要重写。

### 2.5 小结：Hertz 在 coze 里承担的角色

| 职责 | 实现 | 证据文件 |
|---|---|---|
| HTTP 服务入口 / 启动 | `server.Default` + netpoll 引擎 | `backend/main.go` |
| 路由树 | `hz` 从 IDL 生成的分组路由（~246 路由） | `backend/api/router/coze/api.go` |
| 中间件责任链 | 鉴权 + 可观测 + i18n（顺序敏感） | `backend/main.go`、`api/middleware/` |
| 跨域 | `hertz-contrib/cors` | `backend/main.go` |
| 流式 LLM 输出 | `hertz-contrib/sse` | `backend/infra/sse/impl/sse/sse.go` |
| 静态资源 + SPA fallback | `r.Static / r.StaticFile / r.NoRoute` | `backend/api/router/register.go` |
| 私有化部署 | 可配 TLS / body size / 监听地址 | `backend/main.go` |

## 3. ★为什么官方选 Hertz（技术 + 生态 + 平台 三类理由）

### 3.1 纯技术动机

1. **netpoll 高并发底座**（确定的设计事实，具体数字待核实）
   Hertz 默认底层是 netpoll，相对标准库 `net/http` 的"一连接一 goroutine + 阻塞 IO"，netpoll 用主动 epoll 事件循环 + 零拷贝 buffer + 内存池，在**高 QPS / 海量连接**下吞吐和内存表现更好。官方 benchmark 宣称 Hertz 在高并发下吞吐显著高于 Gin、延迟（尤其 P99 长尾）更低——**具体倍数/QPS 数字待核实**，不同压测口径差异很大，关键不是记数字，而是机制："它优化的是高并发长尾，机制是 netpoll 的 epoll 多路复用 + nocopy buffer"。

2. **大 body / 流式友好**
   coze 要传 200MB 知识库文件、要 SSE 流式吐 token。Hertz 在 body 读取、流式响应上有 netpoll 的 nocopy 支撑，且 `hertz-contrib/sse` 是一等公民。这对 AI 平台是刚需，不是锦上添花。

3. **代码生成工具链 hz**
   `hz` 把 IDL → 路由 + handler 骨架 + model 一键生成，246 条路由全自动维护（改 IDL 重新生成即可），人只填中间件和业务体。对一个有几十个 service、上百路由的大平台，这种**契约驱动 + 代码生成**显著降低手写路由的出错率和维护成本。这是 Gin 等"纯手写路由"框架没有的工程化能力。

### 3.2 生态 / dogfooding 动机（部分确定、部分推断）

1. **CloudWeGo 全家桶协同（确定事实）**
   coze 同时用了 Hertz（HTTP）和 Eino（LLM 编排），两者同属 CloudWeGo、同一团队、共享 netpoll 与设计哲学。选 Hertz 让"HTTP 入口 + LLM 编排 + 底层网络"在同一生态里，依赖管理、版本协同、问题排查都顺。

2. **dogfooding（强推断，动机层面）**
   字节是 CloudWeGo 的**出品方**。coze 作为字节自家产品，用自家框架既是"自己人吃自己的狗粮"（验证框架、反哺迭代），也是组织内部的**技术栈统一**——内部基建、培训、SRE 经验、监控接入大概率都是围绕 CloudWeGo 建的，用别的框架反而是"逆生态"。**注意**：这是基于"字节出品 + 自家产品"的合理推断，不是源码能证明的事实，准确表述：源码能确认的只是它用了 Hertz+Eino 同源生态，"生态归属是主因"属合理推断。

3. **超大规模生产验证（推断，数字待核实）**
   CloudWeGo 官方宣称 Hertz 已在字节内部承载海量服务/超大流量。"经过字节内部超大规模验证"是常见且合理的论据，但**具体承载多少服务、多少 QPS 属于待核实**，别引用具体数字。

### 3.3 平台 / 私有化动机（确定事实）

coze-studio 是要**私有化部署给上万企业**的平台，不是一个跑在字节自己机房的内部服务。这带来几个 Hertz 契合的点：

1. **资源效率 = 私有化成本**：企业自己掏机器跑 coze，单机能扛的并发越高、内存占用越低，部署成本越低。netpoll 的资源效率直接转化为"客户少买几台机器"。
2. **可配置的部署面**：可配 TLS（企业自带证书）、监听地址、body 上限——`main.go` 里全有，这是平台型产品的标配。
3. **一个二进制 + 静态资源**：`register.go` 里直接 `r.Static` 托管前端 + `r.NoRoute` 做 SPA fallback，前后端打成一个 Go 服务交付。Go 的单二进制 + Hertz 自带静态托管，对私有化交付（客户环境千奇百怪）非常友好。

> **三类动机的诚实排序**：对 coze 这种字节自家 + CloudWeGo 已有 Eino 在场的项目，**生态归属/dogfooding 是第一驱动**（既然 Eino 是 CloudWeGo 的、字节又是出品方，HTTP 层自然选同生态的 Hertz）；**纯技术（netpoll 性能）是强支撑但非唯一理由**（Gin 也能跑 AI 平台，只是上限低些）；**私有化的资源效率是放大器**（让性能优势变成客户的真金白银）。

## 4. ★同类替代逐个对比 + 为什么不选

对比维度统一用：**定位 / 底层网络 / API 兼容性 / 生态契合度 / 性能取向 / 选错的代价**。性能"倍数"类数字全部标 **待核实**——压测口径差异极大，理解机制比记数字更重要。

### 4.1 Gin —— 最流行，但底层是 net/http

- **定位**：Go 生态事实标准 HTTP 框架，社区最大、文档最多、中间件最全。
- **底层网络**：标准库 `net/http`（一连接一 goroutine + 阻塞 IO + runtime netpoller）。
- **API**：单参数 `func(*gin.Context)`，`c.JSON / c.Param / c.Query`。
- **生态契合度（对 coze）**：低。Gin 不属于 CloudWeGo，和 Eino 没有同源关系，dogfooding 价值为零。
- **性能取向**：路由匹配（radix tree）本身很快，瓶颈在**底层 net/http 的高并发上限**——海量连接下 goroutine 数量和内存拷贝成为天花板。Hertz 用 netpoll 顶掉的正是这一层（**具体差距数字待核实**）。
- **选错的代价**：① 失去 netpoll 在超高并发/长尾延迟上的优势，私有化客户要多买机器；② 失去 CloudWeGo 生态协同（要自己把 Gin 和 Eino 粘起来）；③ 失去 `hz` 的 IDL→路由代码生成，246 条路由要么手写要么自己造轮子。**但反过来说**：Gin 是"安全牌"，团队熟、招人快、踩坑少——如果项目不是字节出品、也没有 Eino 在场、并发量也不极端，选 Gin 完全合理。coze 不选它，主要输在**生态归属**而非"Gin 不行"。

### 4.2 Echo —— Gin 的同类竞品

- **定位**：和 Gin 几乎同生态位的 HTTP 框架，API 略更"面向接口"、内置功能（如 TLS 自动证书、middleware 集）稍丰富。
- **底层网络**：同样 `net/http`。
- **生态契合度（对 coze）**：低，理由同 Gin（非 CloudWeGo）。
- **性能取向**：与 Gin 同量级，天花板都在 net/http。
- **选错的代价**：和 Gin 基本一致，且 Echo 社区比 Gin 小，对 coze 没有任何相对 Gin 的额外优势。**纯粹是"既不如 Gin 流行、又不如 Hertz 同生态"，两头不靠**。

### 4.3 Fiber（fasthttp）—— 性能党，但不兼容 net/http

- **定位**：主打性能、API 模仿 Express（Node.js 风格），底层是 `fasthttp`。
- **底层网络**：`fasthttp`——和 netpoll 一样**绕开标准库**追求高性能，但路线不同（fasthttp 也用连接复用 + 减少分配）。
- **API 兼容性**：**这是 Fiber 最大的坑**——fasthttp **不兼容标准库 `net/http` 的 `Handler` 接口**，它有自己的 `RequestCtx`。后果是：海量基于 `http.Handler` 的标准中间件（OpenTelemetry、pprof、各种 SDK 的 http 中间件）**不能直接用**，必须找 fasthttp 适配版或自己改写。生态是**割裂**的。
- **生态契合度（对 coze）**：低且更糟——既不属于 CloudWeGo，又因为 fasthttp 的不兼容性导致它**连标准 net/http 生态都享受不全**。
- **性能取向**：高，但和 Hertz/netpoll 谁更强**没有公允定论（待核实）**，且对 coze 来说"够快"早就满足，性能不是选它的理由。
- **选错的代价**：拿到一点（不确定的）性能，付出**生态割裂 + 接 Eino 还是要自己粘 + 没有 hz 工具链**的代价。对 coze 是明显的负收益。值得注意的是：Hertz 和 Fiber 走的是同一条"绕开 net/http"的路，但 Hertz 通过自己的抽象层 + hertz-contrib 把生态补回来了，而 Fiber 把生态债留给了使用者。

### 4.4 net/http 标准库 —— 不造框架，自己拼

- **定位**：Go 自带，零依赖，最稳定，最不会"被框架绑架"。
- **底层网络**：标准库自身（一连接一 goroutine）。
- **生态契合度**：中性——任何 `http.Handler` 中间件都能用，但要自己接 Eino、自己拼。
- **性能取向**：足够好，但高并发上限不如 netpoll；且 Go 1.22 后标准库路由（`http.ServeMux` 支持 method + path 模式）增强了不少，小项目其实够用。
- **选错的代价**：对 coze 这种 246 路由 + 鉴权链 + i18n + SSE + 代码生成需求的大平台，**纯标准库 = 大量造轮子**：路由分组、中间件链、参数绑定、IDL 生成全要自己写或拼第三方库，工程效率和一致性都崩。标准库适合**库/工具/小服务**，不适合大型业务平台。

### 4.5 go-zero —— 全家桶式微服务框架

- **定位**：一个**大而全**的微服务框架（不只是 HTTP），自带 RPC、服务发现、限流熔断、`goctl` 代码生成、缓存方案等，"开箱即用一整套微服务治理"。
- **底层网络**：基于标准库 `net/http`（HTTP 部分）。
- **生态契合度（对 coze）**：低且**理念冲突**——go-zero 是"全家桶绑定"，你用它就基本进了它的整套治理体系；而 coze 已经选定 CloudWeGo 生态（Eino），再叠一个 go-zero 的全家桶是**两套微服务哲学打架**。
- **性能取向**：HTTP 层在 net/http 量级，治理能力强但 netpoll 级别的网络性能不是它的卖点。
- **选错的代价**：引入一整套你并不需要的治理体系（coze 开源版是单体，根本不需要 go-zero 的服务发现/RPC 那套），还和已选的 CloudWeGo 生态重复/冲突。**适用场景明显错配**——go-zero 适合"从零搭一套国产微服务治理"，不适合"已经押注 CloudWeGo 的项目"。

### 4.6 Kratos（B 站，gRPC 向）—— 面向 RPC 的微服务框架

- **定位**：B 站开源的微服务框架，**强 gRPC 取向**、强调 API 优先（protobuf）、依赖注入（wire）、整洁分层。
- **底层网络**：HTTP 部分基于 net/http；主打的是 gRPC + HTTP 双协议。
- **生态契合度（对 coze）**：低——Kratos 自成一套生态（B 站系），和 CloudWeGo 没关系；且它的重心是 gRPC 微服务，而 coze 开源版根本不需要 RPC（前面已确认无 Kitex、是单体 HTTP）。
- **选错的代价**：为了一个**用不到的 gRPC/微服务治理重心**引入 Kratos，还要放弃 CloudWeGo 同源协同。**杀鸡用牛刀且牛刀还不顺手**。
- **注意**：Kratos 的真正对位其实是 **Kitex（RPC）**，不是 Hertz（HTTP）。拿它和 Hertz 比，本身就说明 coze 的需求是 HTTP 而非 RPC——这反过来解释了为什么 coze 选 Hertz 而非任何 RPC 框架。

### 4.7 Beego —— 老牌全栈 MVC 框架

- **定位**：早期国产全栈 Web 框架（MVC、ORM、自带工具链），更接近"Go 版的 Django/Rails"。
- **底层网络**：net/http。
- **生态契合度 / 现状**：低且**偏过时**——Beego 的活跃度和社区动能这些年明显下滑，MVC 全栈范式在"前后端分离 + IDL 契约"的现代后端里也不主流。
- **选错的代价**：技术债 + 招人难 + 范式过时，对一个 2025 年的新平台几乎没有任何选它的理由。**纯历史选项，列出来是为了完整性**。

### 4.8 对比速查表

| 框架 | 底层网络 | net/http 接口兼容 | 代码生成 | CloudWeGo 同源 | 对 coze 的核心短板 |
|---|---|---|---|---|---|
| **Hertz** | netpoll | 是（自带抽象层） | hz（IDL→路由） | ✅ | —（本命选择） |
| Gin | net/http | 是 | 无 | ❌ | 并发上限 + 非同源 + 无代码生成 |
| Echo | net/http | 是 | 无 | ❌ | 同 Gin，且社区更小，两头不靠 |
| Fiber | fasthttp | **否** | 无 | ❌ | 生态割裂 + 性能优势不确定 |
| net/http | 标准库 | —（即本体） | 无 | ❌ | 大平台要造大量轮子 |
| go-zero | net/http | 是 | goctl | ❌ | 全家桶与 CloudWeGo 哲学冲突 |
| Kratos | net/http/gRPC | 是 | protoc 系 | ❌ | 重心是 gRPC，coze 用不到 RPC |
| Beego | net/http | 是 | bee | ❌ | 范式过时 + 社区衰退 |

> 性能"倍数/QPS"列**故意不写**——这类数字随压测口径剧烈波动，全部标 **待核实**。关键是理解机制（netpoll vs net/http 的并发模型差异），而非记死数字。

## 5. 适用边界 / 什么时候该换

Hertz 不是银弹。诚实地列出**它不占优甚至不该用**的场景：

1. **小项目 / 内部工具 / 并发量平平**：Gin 或 Go 1.22+ 标准库 `net/http` 完全够用，且团队更熟、招人更易、踩坑更少。Hertz 的 netpoll 优势在中低并发下体现不出来，反而引入了一个相对小众的栈。**"够用就好"时该用 Gin/标准库。**

2. **团队没有 CloudWeGo 经验、也不打算用 Eino/Kitex**：Hertz 最大的隐性收益是生态协同。如果你既不用 Eino 也不用 Kitex，那 Hertz 相对 Gin 的优势就只剩"netpoll 性能"——这点收益是否值得引入一个团队不熟的栈，要权衡。**孤立用 Hertz（不配套 CloudWeGo 其他组件）性价比会下降。**

3. **强依赖某些只有 net/http 生态才有的成熟中间件/SDK**：虽然 Hertz 有自己的 hertz-contrib 和抽象层，但 Go 社区海量中间件是面向 `http.Handler` 的。如果你的关键依赖只有 net/http 版本、且没有 Hertz 适配，迁移成本要算进去。（不过 Hertz 这点比 Fiber 好很多——Hertz 有意做了兼容抽象。）

4. **以 RPC / 微服务治理为核心诉求**：如果你要的是服务间 RPC、服务发现、强治理，那对位的是 **Kitex / go-zero / Kratos**，不是 Hertz。Hertz 只解决 HTTP 入口。

**什么时候该从 Hertz 换走**：团队彻底放弃 CloudWeGo 生态、并发量长期处于低位、且团队/社区里 Hertz 人才稀缺导致维护困难——三者同时成立时，回退到 Gin 是理性的。但只要你还在用 Eino（像 coze、像 vibe-studio），换走 Hertz 就是自找生态割裂。

## 6. 对我们 vibe-studio 的启示（一致性说明）

vibe-studio 的定位是"对齐 coze 但 right-size"。在 Hertz 这件事上，我们**保持一致**，证据在 `/Users/mac/vibe-studio/backend`：

- `backend/go.mod`：
  - `github.com/cloudwego/hertz v0.10.5`（比 coze 的 v0.10.2 略新，同一 minor 线）
  - `github.com/hertz-contrib/cors v0.1.0`（与 coze 完全一致）
  - `github.com/cloudwego/netpoll v0.7.3`（indirect，和 coze 一样由 Hertz 间接引入）
- `backend/cmd/server/main.go`：`h := server.Default(server.WithHostPorts(cfg.HTTPAddr))`，并注释了"server.Default 自带 recover 中间件"——和 coze 的 `server.Default(opts...)` 同款用法。
- 路由同样走 **hz 代码生成**：`backend/api/router/register.go`、`backend/api/router/workflow/workflow.go` 头部都是 `// Code generated by hertz generator. DO NOT EDIT.`，handler `backend/api/handler/workflow/workflow_service.go` 沿用 hz 骨架 + 手动接 application service。

**对齐的理由（和第 3 节同构）**：vibe-studio 也用 CloudWeGo 生态做 LLM 编排，HTTP 层选 Hertz 保证"同源生态"，并直接复用 coze 的 IDL 契约 + hz 工具链经验，迁移/对照成本最低。

**right-size 的体现**：coze 有 246 条路由、8 层中间件、TLS/200MB body/i18n 全套私有化配置；vibe-studio 不需要把这些一次性全上——我们保留 Hertz + netpoll + hz 这条主干（核心价值），但中间件链和路由规模按自己的需求裁剪，不为"对齐"而堆砌用不到的私有化开关。**这正是 "对齐 coze 但 right-size" 的精确含义：选型一致（同生态、同工具链），规模量体裁衣。**
