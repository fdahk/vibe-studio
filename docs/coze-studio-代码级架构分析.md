## 0. 速览（先建立体感）

| 指标 | 实测值 |
|---|---|
| 后端 Go 文件数 | **1013** |
| `api/model` 代码量 | **~49.3 万行**，全部 `// Code generated by thriftgo (0.4.2). DO NOT EDIT.` |
| Eino 被引用的 go 文件数 | **205**（workflow 引擎 65 + agent 19 + 文档解析 18 + …） |
| 架构风格 | **DDD 洋葱/六边形**（api→application→domain←infra） |
| 依赖注入 | **手写 DI**（无 wire），`application.Init` 四层显式初始化 |
| 代码生成 | **Thrift IDL → thriftgo（model）+ hz（main/router）**；`main.go` 头部写着 `// Code generated by hertz generator.` |
| 数据库迁移 | **Atlas**（`docker/atlas` + Makefile `sync_db/dump_db`） |

**一句话**：这是一套"**IDL 契约先行 + 代码生成 + DDD 分层 + Eino 编排 + 全套可插拔基础设施**"的工业级 Go 后端。**手写的核心是业务逻辑（domain/application/handler），大量样板（model/router）是生成的。**

## 1. 顶层项目组织（monorepo）

```
coze-studio/
├── backend/      Go 后端（本文主角，微服务化 DDD）
├── frontend/     React+TS 大型 monorepo（Rush + pnpm + Rsbuild，400+ 包，占代码 80%）
├── common/       前后端/多模块共享
├── idl/          ★ Thrift 接口定义（契约先行，驱动代码生成）
├── docker/       部署：docker-compose 多变体 + nginx + atlas(迁移) + volumes(schema/es)
├── helm/         Kubernetes 部署（helm charts/opencoze）

├── Makefile      构建与运维编排
└── rush.json     前端 monorepo 配置
```

`idl/` 的组织（按业务域切 thrift）：
```
idl/api.thrift  base.thrift
idl/app/(developer_api/project/publish/intelligence/bot_open_api...)  playground/  marketplace/
idl/workflow/  plugin/  conversation/  data/  permission/  resource/  upload/  admin/  passport/
```
**IDL 先行**是整个工程的"地基契约"——前后端、各服务都按这套 thrift 对齐，再生成代码。

### 2.2 代码生成链（这是理解 coze 的关键）

```
idl/*.thrift
   │ thriftgo (0.4.2)
   ├─→ backend/api/model/**        // 49 万行 struct，DO NOT EDIT
   │ hz (hertz generator)
   ├─→ backend/main.go             // 头部: // Code generated by hertz generator.
   ├─→ backend/api/router/**       // router.GeneratedRegister(s)
   └─→ backend/api/handler/** 的 stub（人再填业务）
```
证据：`api/model` 首行 `// Code generated by thriftgo (0.4.2). DO NOT EDIT.`；`main.go:17` `// Code generated by hertz generator.`；`api` 层 50 万行里 **model 49.3 万 / handler 1 万 / router 2.4k**——可见**绝大部分是生成的样板**。

## 3. 后端分层架构（DDD 洋葱/六边形）

```
            ┌───────────────────────────────────────────────┐
   入站 →   │ api/        handler · router · model · middleware │ (Hertz, 多为生成)
            └───────────────────┬───────────────────────────┘
            ┌───────────────────┴───────────────────────────┐
            │ application/   用例编排(每个域一个 ApplicationService) │
            └───────────────────┬───────────────────────────┘
            ┌───────────────────┴───────────────────────────┐
            │ crossdomain/   跨域协调(把 domain 暴露成全局 SVC)     │
            └───────────────────┬───────────────────────────┘
            ┌───────────────────┴───────────────────────────┐
   最纯 →   │ domain/        实体 + 领域服务 + repository 接口      │ (业务核心)
            └───────────────────▲───────────────────────────┘
            ┌───────────────────┴───────────────────────────┐
            │ infra/         repository 实现 + 中间件客户端        │ (依赖倒置)
            └───────────────────────────────────────────────┘
   横切: pkg/(errorx/logs/safego/...) · types/(consts/errno/ddl) · bizpkg/(config/llm/...)
```

各层目录清单（实测）：
- **domain/**（核心域）：`agent / app / connector / conversation / datacopy / knowledge / memory / openauth / permission / plugin / prompt / search / shortcutcmd / template / upload / user / workflow`
- **application/**（用例）：与 domain 基本一一对应，外加 `base / modelmgr / singleagent / ...`
- **crossdomain/**：`agent / agentrun / app / conversation / database / knowledge / message / plugin / variables / workflow / ...`（解决"A 域要调 B 域"）
- **infra/**：`orm / rdb / cache / eventbus / storage / es / embedding / idgen / sse / checkpoint / coderunner / document / imagex / dynconf / oceanbase / sqlparser`
- **pkg/**：`errorx / logs / safego / taskgroup / ctxcache / i18n / hertzutil / sonic / ...`

依赖方向严格**从外向内**，`domain` 不依赖框架；`infra` 实现 `domain` 定义的 repository 接口（**依赖倒置 DIP**）。

### 4.2 `application.Init` —— 手写 DI 的四层初始化

`backend/application/application.go`（约 124–171 行）是整个系统的**组合根**，分层初始化避免循环依赖：

```go
func Init(ctx context.Context) (err error) {
    ctx = ctxcache.Init(ctx)                                  // 请求级缓存
    infra, err := appinfra.Init(ctx)                          // ① 基础设施: DB/Redis/ES/OSS/IDGen/CodeRunner/Parser...
    basicServices, err := initBasicServices(ctx, infra, eventbus)   // ② 只依赖 infra: permission/connector/user/prompt
    primaryServices, err := initPrimaryServices(ctx, basicServices) // ③ 依赖 basic: plugin/memory/knowledge/workflow
    complexServices, err := initComplexServices(ctx, primaryServices)// ④ 依赖 primary: singleagent/app/search/conversation
    // ⑤ 把 domain service 注册成全局单例，供跨域调用：
    crosspermission.SetDefaultSVC(permissionImpl.InitDomainService(...))
    crossworkflow.SetDefaultSVC(workflowImpl.InitDomainService(...))
    // ... 共 ~11 个 domain 通过 crossdomain 的 SetDefaultSVC 全局化
}
```

**设计要点**：① 无 wire，**显式 Init 链**，依赖关系一眼可见、顺序可控；② 用 `crossdomain/*` 的 `SetDefaultSVC()` 把领域服务挂成全局单例，解决"跨域互调"又不破坏分层。

## 5. API 层与 IDL 代码生成——代码级

一个请求的完整下沉链路（以 conversation/message 为例）：

```go
// ① api/handler/coze/message_service.go:34  —— 手写 handler
func GetMessageList(ctx context.Context, c *app.RequestContext) {
    var req message.GetMessageListRequest  // ← thriftgo 生成的结构体(api/model)
    c.BindAndValidate(&req)
    resp, err := application.ConversationSVC.GetMessageList(ctx, &req) // 调用应用层

}
// ② application/conversation/init.go:46 —— 应用层组装 domain
mDomainComponents := &message.Components{
    MessageRepo: msgRepo.NewMessageRepo(s.DB, s.IDGen),  // repo 注入 infra.DB
}
messageDomainSVC := message.NewService(mDomainComponents)
// ③ domain/conversation/message/service —— 领域服务接口
```

## 7. 核心子系统深剖：Workflow 执行引擎（最重要）

目录：`backend/domain/workflow/`，其中 `internal/` 是引擎本体。

### 7.1 数据流全链路

```
前端 Canvas(JSON, vo.Canvas)
  │ ① 校验  internal/canvas/validate/canvas_validate.go (NewCanvasValidator) —— 连通性/可达性/变量引用
  │ ② 转换  internal/canvas/adaptor/to_schema.go (CanvasToWorkflowSchema) —— 剪枝孤立节点/批量展开/端口归一/构建分支

  │ ③ 编排  internal/compose/workflow.go (NewWorkflow) —— 转成 Eino Workflow+Graph，逐节点 Build+注册
  │ ④ 编译  wf.Compile(ctx) → compose.Runnable
  ▼
执行  Runner.Invoke()/Stream()  internal/compose/workflow_run.go + internal/execute/
```

关键代码：

### 7.2 如何用 Eino（编排内核）

用的是 Eino 的 **`compose.Workflow[map[string]any, map[string]any]` + Graph**：
```go
// internal/compose/workflow.go:37
type workflow = compose.Workflow[map[string]any, map[string]any]
type Workflow struct {
    *workflow
    Runner compose.Runnable[map[string]any, map[string]any] // 编译后可运行对象
}

// internal/compose/workflow.go:83
func NewWorkflow(ctx, sc *schema.WorkflowSchema, opts ...) (*Workflow, error) {
    wf := &Workflow{ workflow: compose.NewWorkflow[map[string]any,map[string]any](
        compose.WithGenLocalState(GenState())) }
    for _, cNode := range sc.GetCompositeNodes() { wf.AddCompositeNode(ctx, cNode) } // Loop/Batch 递归内层
    for _, ns := range sc.Nodes { wf.AddNode(ctx, ns) }            // 普通节点
    if wf.requireCheckpoint {
        compileOpts = append(compileOpts, compose.WithCheckPointStore(GetRepository())) // 注入 checkpoint
    }
    wf.Runner, _ = wf.Compile(ctx, compileOpts...)                  // Eino 编译成 Runnable
}
```
节点注册三步（`internal/compose/workflow.go` addNodeInternal）：① `resolveDependencies` 算字段映射 → ② `New(...)` 实例化节点（每个节点最终是一个 `compose.Lambda`）→ ③ `AddNode` 并挂 `preHandler/postHandler`（状态保存/结果处理钩子）。
运行分流式/同步：`Runner.Stream(ctx, in)`（逐 item）/ `Runner.Invoke(ctx, in)`（等结果）。

### 7.3 节点体系（`internal/nodes/`，20 类）

```
入出口: entry  exit  receiver  emitter
通用:   llm  code  httprequester  json  textprocessor  database
控制流: selector(条件分支)  loop(循环, 含 break/continue)
聚合:   variableaggregator  variableassigner  batch
AI:     knowledge(RAG)  qa  intentdetector  plugin
组合:   subworkflow  conversation
```
4 个代表：
| 节点 | 文件 | 实现要点 |
|---|---|---|
| LLM | `nodes/llm/llm.go` | 加载模型+Prompt 模板；底层接 Eino react agent，支持 tool calling/流式/多轮 |
| Code | `nodes/code/code.go` | Build 期做 Python import 黑/白名单校验；Invoke 调 `coderunner.Run` |

### 7.6 checkpoint / 中断恢复

```go
// infra/checkpoint/redis.go:30
const checkpointExpire = 24*7*3600*time.Second   // 7 天 TTL
func (r *redisStore) Set(ctx, id string, cp []byte) error {
    return r.client.Set(ctx, fmt.Sprintf("checkpoint_key:%s", id), cp, checkpointExpire).Err()

// internal/compose/state.go: init() 里 compose.RegisterSerializableType[*State]/[vo.NodeKey]/[*execute.Context]...
// 编译时 compose.WithCheckPointStore(redisStore) 注入 → Eino 在节点后自动存 checkpoint，可 Resume 续跑
```
即：**Eino 负责"在哪存、何时存"，coze 提供"存到 Redis"的实现 + 注册可序列化类型**。`mem.go` 是内存实现供测试。

### 7.7 Code 节点沙箱（与我们最相关）

```go
// infra/coderunner/code.go —— 统一接口
type Runner interface { Run(ctx, *RunRequest) (*RunResponse, error) }
type RunRequest struct { Code string; Params map[string]any; Language Language } // Python / JavaScript
```
两种实现：
- `impl/direct/runner.go`：**直接 exec**（模板注入用户代码后跑），无强隔离——开发/可信场景。
- `impl/sandbox/runner.go` + `impl/script/sandbox.py`：**沙箱**，经 Python 脚本受限执行（配合 `make python` 装的环境）。
节点侧再加一道防线：`nodes/code/code.go` 在 Build 期校验 Python import（黑名单含 `socket/threading/multiprocessing/...`），不合规直接拒跑。

## 8. 其他用到 Eino 的子系统（简述）

- **agent/singleagent**（19 文件用 Eino）：单 agent 运行时 = Eino react agent + tool calling。
- **conversation/agentrun**：会话级 agent 执行编排。
- **knowledge/RAG**：`infra/document/parser`（文档解析）、`searchstore`（检索）、`embedding/impl`（向量化）、`nl2sql` 都用 Eino 组件串管线。
- **bizpkg/llm/modelbuilder**：模型抽象与构建（多家模型统一成 Eino model）。

> 结论：**Eino 是横跨 workflow / agent / 知识库 的统一编排底座**，不是某一处的局部选型。

## 9. 配置与横切能力

- **配置**：`godotenv` 读 `.env*`；`backend/conf/*.yaml`（`model/plugin/prompt/workflow`）+ DB 动态配置；`bizpkg/config/config.go:Init(ctx, db, oss)` 把 DB+OSS 里的模型/知识库配置加载成进程内 `shardConfig`。
- **pkg 横切**：`errorx`（带 status code/extra/堆栈的错误）、`logs`（分级日志）、`safego`（带 recover 的 goroutine）、`taskgroup`（基于 errgroup 的并发组）、`ctxcache`（请求级 `context`+`sync.Map` 缓存）、`i18n`（en-US/zh-CN）。

## 10. 端到端数据流（把全文串起来）

```
HTTP 请求
  → Hertz 中间件链(ctxcache→inspector→host→logid→cors→accesslog→openapiAuth→sessionAuth→i18n)
  → router.GeneratedRegister 路由(生成) → api/handler(手写)
  → 绑定 thriftgo 生成的 req → application.XxxSVC(用例编排)
  → domain service(业务规则) → repository 接口
  → infra 实现(GORM/Redis/ES/MinIO/Milvus/eventbus)
若是"运行工作流":
  application/workflow → domain/workflow: Canvas校验→adaptor转Schema→compose(Eino Graph)→Compile→Runner.Stream/Invoke
     节点执行: 每节点 Lambda；字段映射注入输入；LLM/Code/Selector/Loop... ;
     流式经 StreamContainer→SSE 推前端；可选 checkpoint 存 Redis 支持中断续跑
```

## 11. 对我们项目（vibe-studio）的启发（要点，详见决策文档）

1. **生成 vs 手写的边界**：coze 把"协议样板(model/router)"交给 IDL+codegen，人只写业务。我们右size 暂不上 Thrift codegen，但**这是它"对齐官方"时最该补的一环**（上 IDL 时用 `hz`）。
2. **手写四层 DI** 值得学：显式 Init 链 + crossdomain 全局 SVC，比一上来就 wire 更易懂。
3. **infra 统一接口 + 多实现 + env 切换**：我们的 `infra/mysql|redis|storage` 已是这个思路的雏形，继续保持。
4. **引擎是 Eino 的二次封装**：Canvas→Schema→Eino Graph→Runnable + 字段映射 + checkpoint。我们**自研引擎**时，这就是"标准答案"——但要自己实现编译/调度，而不是直接用 Eino（否则深核不是你的）。
5. **Code 沙箱 direct/sandbox 双实现 + import 黑白名单**：正好对应我们技术点02/08 的沙箱设计，可直接借鉴这套"接口 + 多隔离级别 + 校验前置"。
